🛠️ Coding Practices & SDLC

This blog was developed following an iterative sprint-based process that mirrors industry SDLC practices. Code quality, documentation, and version control were maintained throughout each sprint.

Sprint-based development: The Peppa Pig Fight Club game was built across Sprint 6, with incremental commits to GitHub, peer code reviews, and iterative playtesting to identify and fix bugs before each release.

Key Coding Practices Applied

  • Separation of concerns: game logic (PeppaBattleLevelBase), entity behavior (PeppaBossEnemy), and configuration (level data objects) are kept in separate classes.
  • DRY (Don't Repeat Yourself): shared laser logic, HUD rendering, and collision detection live in the base class so all three levels inherit them without duplication.
  • Defensive programming: optional chaining (?.) and nullish coalescing (??) guard against null/undefined config values that would crash the level.
  • Cleanup in destroy(): event listeners are removed and timers cleared when a level ends, preventing memory leaks in long gameplay sessions.
  • Console logging strategy: every state-changing operation logs a prefixed message ([Boss], [Hitbox], [Leaderboard]) to make runtime debugging fast and readable.

SDLC — Software Development Life Cycle

PhaseWhat Was Done
Planning Defined the 3-level boss fight structure, identified required classes, and mapped rubric requirements to specific code features before writing any game logic.
Design Sketched the class hierarchy (GameObject → Enemy → PeppaBossEnemy, PeppaBattleLevelBase → PeppaLevel1/2/3), designed the laser data structure, and decided on the AABB collision algorithm.
Implementation Built classes incrementally — boss constructor first, then movement, then damage system, then laser spawning, then HUD, then win/lose conditions. Each feature was committed separately.
Testing Play-tested each level end-to-end, used DevTools console + hitbox visualizer to debug collision issues, and verified the leaderboard API with Network tab monitoring.
Deployment Published via GitHub Pages through the OCS GameEngine pipeline. Each sprint push triggered automatic deployment to the live student site.
Maintenance Fixed hitbox sizing after playtesting, tuned boss speeds per level, and adjusted the invulnerability window from 200ms to 300ms based on gameplay feel.
Built with JavaScript, Canvas API, and the OCS GameEngine
